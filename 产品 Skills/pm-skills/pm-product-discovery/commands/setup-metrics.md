# /setup-metrics -- 产品指标看板设计

为你的产品或功能构建一套完整的指标体系——从选定正确的北极星指标，到定义能早期发现问题的告警阈值。

## 调用方式

```
/setup-metrics SaaS 项目管理工具
/setup-metrics 我们刚上线的新结账流程
/setup-metrics             # 询问你要衡量什么
```

## 工作流程

### 第一步：理解衡量目标

询问用户：
- 你要为哪个产品或功能方向搭建指标体系？
- 目前处于哪个阶段？（上线前、刚上线、成熟期）
- 当前的业务目标或 OKR（目标与关键结果）是什么？
- 是否已有现有指标？有哪些缺失或存在问题？
- 使用哪些分析工具？（有助于提供针对性的实施建议）

### 第二步：定义指标框架

应用 **metrics-dashboard** 技能：

**北极星指标：**
- 识别最能捕捉产品为用户创造价值的单一指标
- 按以下标准验证：衡量价值交付、是领先指标、可采取行动
- 精确定义指标（计算公式、数据来源、时间窗口）

**输入指标（3-5 个）：**
- 识别驱动北极星指标的关键杠杆
- 每个输入指标应可由具体团队直接影响
- 绘制因果链：输入指标 → 北极星指标 → 业务产出

**健康指标（3-5 个）：**
- 应保持稳定的指标——一旦下降，说明出现了问题
- 示例：错误率、响应延迟、支持工单量、NPS（净推荐值）、流失率
- 定义"健康"区间和下降告警阈值

**反向指标（1-2 个）：**
- 用于识别是否在朝错误方向优化的指标
- 示例：如果北极星指标是"日活用户数"，反向指标是"会话质量"，以防止空洞的活跃度刷数

### 第三步：设计告警阈值

为每个指标设定：

| 指标 | 绿色（正常） | 黄色（警告） | 红色（告警） | 检查频率 |
|------|------------|------------|------------|--------|
| [指标] | [健康区间] | [需关注] | [需立即处理] | [每日/每周] |

- **黄色**：调查——可能出现了异常
- **红色**：立即行动——通知相关人员或上报

### 第四步：创建看板规格文档

```
## 指标看板：[产品/功能]

**北极星指标**：[指标名称]
**定义**：[精确计算公式]
**当前值**：[如已知]
**目标值**：[目标]

### 输入指标
| 指标 | 定义 | 负责人 | 目标值 | 当前值 |
|------|------|--------|--------|--------|

### 健康指标
| 指标 | 健康区间 | 黄色阈值 | 红色阈值 |
|------|----------|----------|----------|

### 反向指标
| 指标 | 重要性 | 需监控的信号 |
|------|--------|------------|

### 指标树
北极星指标：[指标]
├── 输入：[指标 1] → 由 [团队/行动] 驱动
├── 输入：[指标 2] → 由 [团队/行动] 驱动
├── 输入：[指标 3] → 由 [团队/行动] 驱动
└── 反向：[指标] → 监控 [下降信号]

### 实施说明
- 数据来源：[每个指标的数据出处]
- 刷新频率：[实时 / 每小时 / 每日]
- 工具推荐：[基于用户的技术栈]

### 评审节奏
- **每日**：快速浏览北极星指标和健康指标
- **每周**：回顾输入指标趋势，在团队站会中讨论
- **每月**：深度分析——输入指标是否如预期驱动北极星指标？
- **每季度**：重新评估指标体系本身
```

保存为 markdown 文件到用户工作区。

### 第五步：提供后续选项

- "需要我**编写 SQL 查询**来计算这些指标吗？"
- "要不要基于这套指标框架**创建 OKR（目标与关键结果）**？"
- "想要我**搭建留存分析**来设定合理的基准值吗？"
- "要不要**建立每周指标回顾模板**？"

## 注意事项

- 好的北极星指标很罕见——大多数团队都选了虚荣指标。要坚持寻找能真正反映*用户价值交付*而非单纯活跃度的指标
- 输入指标应满足 MECE 原则（相互独立、完全穷尽），共同解释北极星指标的变化
- 如果产品尚未上线，现在就定义指标，但请注意基准值需要在上线后校准
- 反向指标能防止古德哈特定律效应——当一个指标变成目标，它就不再是好指标了
- 建议从少量、仪器完善的指标开始，而非建一个没人看的大而全看板
