# /triage-requests -- 功能需求分流处理

将来自支持工单、销售通话、用户调研或 Slack 消息中堆积的功能需求，整理成一份经过优先级排序、可落地执行的待办列表。

## 调用方式

```
/triage-requests                           # 询问输入内容
/triage-requests [粘贴需求列表]
/triage-requests [上传 CSV/表格文件]
```

## 工作流程

### 第一步：接收功能需求

接受任意格式：
- **粘贴文本**：需求列表，每行或每段一条
- **上传文件**：包含需求数据的 CSV、Excel 或文本文件
- **结构化数据**：如果输入包含列（提需求方、需求内容、日期等），保留其结构

如果未提供输入，请用户粘贴或上传功能需求。

解析每条需求，提取：
- 核心诉求（用户想要什么）
- 背景信息（谁提的、何时提的、为什么——如有）
- 频率信号（有多少人提了类似需求）

### 第二步：收集优先级排序背景

向用户提问（对话式展开，不要一次性抛出所有问题）：
- 产品是什么？处于哪个阶段？
- 当前的战略目标或 OKR（目标与关键结果）是什么？（有助于评估战略契合度）
- 有哪些约束条件？（团队规模、技术债、即将到来的截止日期）
- 是否有需要赋予更高权重的用户细分群体？（企业用户、即将流失的用户、深度用户）

### 第三步：分类与分析

应用 **analyze-feature-requests** 技能：

- **主题聚类**：将相似需求归类为主题（如"报表与分析"、"协作功能"、"移动端体验"）
- **每个主题的需求数量**：映射到该主题的独立需求条数
- **战略契合度**：对照既定目标评估每个主题（高/中/低/无）
- **细分分析**：哪些用户群体在推动哪些主题
- **情绪信号**：需求是否伴随着挫败感、流失威胁或满意表达？

### 第四步：优先级排序

应用 **prioritize-features** 技能：

对每个主题（及主题内的头部个体需求）进行评估：

| 维度 | 评估内容 |
|------|----------|
| **影响力** | 影响多少用户？影响有多严重？ |
| **战略契合度** | 是否服务于当前目标？ |
| **工作量估算** | T 恤尺码（S/M/L/XL） |
| **风险** | 不做这个会怎样？ |
| **收入信号** | 是否与成单、留存或扩展收入相关？ |

对主题排序并生成优先级列表。

### 第五步：生成分流报告

```
## 功能需求分流报告

**日期**：[今日日期]
**已分析需求数**：[数量]
**识别主题数**：[数量]

### 主题汇总
| # | 主题 | 需求数 | 核心诉求 | 战略契合 | 影响力 | 工作量 | 优先级 |
|---|------|--------|----------|----------|--------|--------|--------|

### 优先级一：立即行动
[近期规划中需纳入的主题/需求]
- **[主题]**：[X] 条需求 — [为什么紧迫]
  - 头部需求：[列表]
  - 推荐行动：[建设 / 原型 / 调查]

### 优先级二：规划下一步
[有价值但不紧迫的主题]

### 优先级三：收集更多信号
[有潜力但证据不足的主题]

### 优先级四：拒绝或搁置
[与战略不符的需求——附说明理由]

### 值得关注的个体需求
[高价值但未能归入主题的单独需求]

### 规律与洞察
- [用户通过需求传递的关键信号]
- [特定细分用户群体的规律]
- [用户显性诉求与潜在需求之间的差距]
```

将报告保存为 markdown 文件到用户工作区。

### 第六步：提供后续选项

- "需要我为高优先级事项**创建用户故事**吗？"
- "要不要针对某些主题**头脑风暴解决方案**？"
- "想要在建设前**设计实验**来验证需求真实性吗？"
- "要不要**起草一份利益相关方沟通摘要**来总结本次分析？"

## 注意事项

- 如果用户提供了带列名的 CSV，保留数据结构并进行补充丰富
- 挖掘需求背后的深层需求——"添加暗黑模式"背后可能是"减少长时间使用时的眼睛疲劳"
- 标注相互冲突的需求（例如"简化界面"与"增加更多配置选项"）
- 如果需求量较大（50 条以上），先汇总主题，再按需深入特定主题
- 如果输入是结构化数据，将补充后的数据输出为可下载的 CSV 文件
