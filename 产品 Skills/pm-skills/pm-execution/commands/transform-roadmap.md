# /transform-roadmap -- 成果导向路线图

将计划功能列表或以产出为中心的路线图，重写为以成果为中心的路线图，传达"为什么做"而非"做什么"。

## Invocation（调用示例）

```
/transform-roadmap [粘贴你的功能列表或路线图]
/transform-roadmap [上传路线图文档、表格或截图]
```

## Workflow（工作流程）

### Step 1：接收当前路线图

接受任意格式：
- 功能列表或待办事项
- 路线图文档（现在/接下来/以后，按季度，时间轴）
- 表格或甘特图导出
- 路线图工具的截图

解析每个条目，提取：功能名称、描述、目标日期/时间窗口和任何背景信息。

### Step 2：理解战略背景

询问：
- 这个时期的产品目标或 OKR（目标与关键成果）是什么？
- 这份路线图面向哪些受众？（高管、工程团队、客户、董事会）
- 你偏好哪种格式？（现在/接下来/以后、按季度、时间轴）

### Step 3：转化每个条目

应用 **outcome-roadmap** 技能：

对路线图上的每个功能/产出条目：
1. 识别其试图实现的**用户或业务成果**
2. 重写为成果描述："为 [用户群体] [动词] [指标/体验]"
3. 将服务于同一成果的多个功能合并为一个计划
4. 为每个成果添加成功指标

**前后对比示例：**
- "构建 SSO 集成" → "降低企业客户入门摩擦——目标：企业账户首次使用时间（TTV）缩短 50%"
- "重新设计仪表板" → "帮助重度用户更快获取洞察——目标：洞察获取时间减少 30%"
- "增加 CSV 导出" → "让团队能够在产品外分享数据——目标：报告分享量增加 25%"

### Step 4：生成转化后的路线图

```
## 成果导向路线图：[产品] — [时间段]

**战略主题**: [2-3 个高层次主题]

### 现在（本季度）
**主题：[战略主题]**
| 成果 | 成功指标 | 关键计划 | 状态 |
|------|---------|---------|------|

### 接下来（下季度）
**主题：[战略主题]**
| 成果 | 成功指标 | 关键计划 | 置信度 |
|------|---------|---------|-------|

### 以后（未来）
**主题：[战略主题]**
| 成果 | 成功指标 | 关键计划 | 依赖项 |
|------|---------|---------|-------|

### 转化说明
| 原始功能 | 转化后成果 | 这样表述的原因 |
|---------|---------|-------------|

### 变化总结
[路线图叙事方式的转变概述]
```

保存为 Markdown 文件。

### Step 5：评审

后续建议：
- "需要我**为每个成果添加 OKR 对齐关系**吗？"
- "需要我**起草这份路线图的干系人演示文稿**吗？"
- "需要我**识别"现在"板块的风险**吗？"

## Notes（注意事项）

- 成果应该是可衡量的，且有清晰的"完成"状态
- 多个功能可以对应同一个成果——这是优点，不是问题
- 如果一个产出不能明确对应某个成果，标注出来让用户说明理由或降低优先级
- 受众很重要：高管看的路线图应该只有成果，工程师看的路线图可以在每个成果下包含交付物
- "以后"板块的条目应更不具体——只有成果，没有承诺的解决方案
